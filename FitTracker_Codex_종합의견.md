# FitTracker 프로젝트 종합 의견 (Codex)

작성일: 2026-03-26  
검토 문서: `FitTracker_Portfolio.docx`, `FitTracker_Interview_Guide.docx`, `FitTracker_Job_Guide.docx` + 사용자 제공 GPT 코멘트

## 1) 최종 평가
이 프로젝트는 신입 지원 포트폴리오에서 **충분히 메인으로 사용 가능한 수준**입니다.  
핵심 강점은 고난도 스택 자체가 아니라, 아래 4가지를 끝까지 연결한 완결성입니다.

- 문제의식 명확함: 주관식 강도 기반 추정 방식에 대한 의문에서 출발
- 데이터 근거 있음: MET, Lytle & Lambert, BMR 이중 공식, 식약처 공공 API
- 서비스 흐름 완성: 운동/식단/기록/시각화/밸런스가 하나의 사용자 흐름으로 연결
- 배포 완료: GitHub Pages + PWA까지 실제 동작 가능한 결과물

## 2) GPT 코멘트에 대한 제 판단
사용자께서 전달하신 GPT 평가 방향은 전반적으로 타당합니다. 특히 아래 포인트는 매우 정확합니다.

- “정확 계산”이 아니라 “연구 기반 추정”으로 포지셔닝해야 한다
- 기술 과장보다 1인 완성 경험을 강조해야 한다
- API 필드 매핑 오류, UTC 버그 해결 사례는 면접용 스토리로 강하다
- 직무별로 강조점(기획/운영 vs 개발)을 다르게 가져가야 한다

추가로 제가 보완하고 싶은 점은 2가지입니다.

- README/문서에서 표현 통일: `개인 맞춤형`보다 `개인 입력 기반 추정`을 기본 표현으로 고정
- 링크 운영 전략 분리: 비공개 개발용 레포와 공개 포트폴리오 레포(정리본) 분리 시 리스크 관리가 더 쉽습니다

## 3) 현재 문서 품질 평가
세 문서 모두 실전 활용도가 높습니다. 특히 좋은 점은 다음과 같습니다.

- 이력서 문구, 1분/3분 스크립트, 예상 질문 대응이 이미 구조화되어 있음
- 과장 금지 문구(정확 계산/완벽 서비스 등) 가이드가 포함되어 있음
- 직무별 어필 포인트가 분리되어 있어 지원서 커스터마이징이 쉬움

즉, 지금 상태는 “초안”이 아니라 **바로 제출 가능한 운영 문서 세트**에 가깝습니다.

## 4) 취업 활용 전략 (실행 우선순위)
1. 이력서에는 4줄 핵심만 유지
- 문제정의 1줄
- 구현/근거 1줄
- API/배포 1줄
- 버그해결 1줄

2. 포트폴리오 PDF/Notion에는 사용자 흐름 스크린샷 추가
- 입력 화면 → 결과 화면 → 기록/차트 화면 순서
- 각 화면에 “왜 필요한 화면인지” 1줄 설명

3. 면접 준비는 답변 암기가 아니라 구조 고정
- 상황 → 본인 행동 → 결과/배운 점(STAR)
- API 0kcal 오류, UTC 날짜 오류 2개는 반드시 즉답 가능 상태로 준비

4. 지원 직무별 첫 문장을 다르게 사용
- 기획/운영: “문제 정의와 사용자 흐름 설계, 공공데이터 연동 중심의 1인 서비스 완성 경험”
- 개발: “Vanilla JS 기반 계산 엔진, REST API 연동, PWA 배포까지 구현한 프론트엔드 프로젝트”

## 5) 면접에서 가장 중요한 한 문장
“저는 기능을 나열한 프로젝트가 아니라, 문제를 정의하고 데이터 근거로 설계한 뒤 실제 배포 가능한 서비스로 끝까지 완성했습니다.”

## 6) 결론
FitTracker는 신입 기준에서 경쟁력 있는 프로젝트입니다.  
이 프로젝트의 승부 포인트는 화려한 기술명이 아니라, **검증 가능한 링크 + 설명 가능한 의사결정 + 실제 문제 해결 경험**입니다.
